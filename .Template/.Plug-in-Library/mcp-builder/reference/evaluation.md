# MCP 서버 평가 가이드

## 개요

이 문서는 MCP 서버를 위한 포괄적인 평가를 만드는 방법을 안내합니다. 평가는 LLM이 제공된 도구만을 사용하여 현실적이고 복잡한 질문에 효과적으로 답할 수 있는지 테스트합니다.

---

## 빠른 참조

### 평가 요구사항
- 10개의 사람이 읽을 수 있는 질문 생성
- 질문은 반드시 **읽기 전용, 독립적, 비파괴적**이어야 함
- 각 질문은 여러 도구 호출 필요 (잠재적으로 수십 번)
- 답변은 단일 검증 가능한 값이어야 함
- 답변은 안정적이어야 함 (시간이 지나도 변하지 않음)

### 출력 형식
```xml
<evaluation>
   <qa_pair>
      <question>질문 내용</question>
      <answer>단일 검증 가능한 답변</answer>
   </qa_pair>
</evaluation>
```

---

## 평가의 목적

MCP 서버의 품질 척도는 서버가 도구를 얼마나 잘 또는 포괄적으로 구현했는가가 아니라, **이 구현(입출력 스키마, 독스트링/설명, 기능)이 MCP 서버에만 접근할 수 있는 LLM이 현실적이고 어려운 질문에 답할 수 있도록 얼마나 잘 지원하는가**입니다.

## 평가 개요

읽기 전용, 독립적, 비파괴적, 멱등적 작업만으로 답할 수 있는 10개의 사람이 읽을 수 있는 질문을 만드세요. 각 질문은 다음을 충족해야 합니다:
- 현실적
- 명확하고 간결
- 모호하지 않음
- 복잡: 잠재적으로 수십 번의 도구 호출 또는 단계 필요
- 사전에 확인한 단일 검증 가능한 값으로 답변 가능

## 질문 가이드라인

### 핵심 요구사항

1. **질문은 반드시 독립적이어야 함**
   - 각 질문은 다른 질문의 답변에 의존하면 안 됨
   - 다른 질문 처리 결과로 발생한 이전 쓰기 작업을 가정하면 안 됨

2. **질문은 비파괴적이고 멱등적인 도구 사용만 필요**
   - 올바른 답변에 도달하기 위해 상태를 수정하도록 지시하거나 요구하면 안 됨

3. **질문은 현실적이고, 명확하고, 간결하고, 복잡해야 함**
   - 다른 LLM이 답하기 위해 여러 도구 또는 단계를 사용해야 함

### 복잡성과 깊이

4. **질문은 심층적인 탐색을 필요로 해야 함**
   - 여러 하위 질문과 순차적 도구 호출이 필요한 다중 홉 질문 고려
   - 각 단계가 이전 질문에서 찾은 정보로부터 이익을 얻어야 함

5. **질문은 광범위한 페이지네이션을 필요로 할 수 있음**
   - 여러 페이지의 결과를 페이지네이션해야 할 수 있음
   - 틈새 정보를 찾기 위해 1-2년 된 오래된 데이터를 조회해야 할 수 있음
   - 질문은 반드시 **어려워야 함**

6. **질문은 심층적인 이해를 필요로 해야 함**
   - 표면적 지식이 아닌 깊은 이해 필요
   - 증거가 필요한 참/거짓 질문으로 복잡한 아이디어를 제시할 수 있음
   - LLM이 다양한 가설을 검색해야 하는 객관식 형식 사용 가능

7. **질문은 단순한 키워드 검색으로 풀 수 없어야 함**
   - 대상 콘텐츠의 특정 키워드를 포함하지 않기
   - 동의어, 관련 개념, 의역 사용
   - 여러 검색, 여러 관련 항목 분석, 컨텍스트 추출, 답변 도출 필요

### 도구 테스트

8. **질문은 도구 반환 값을 스트레스 테스트해야 함**
   - 대용량 JSON 객체나 목록을 반환하는 도구를 유발하여 LLM을 압도할 수 있음
   - 여러 형태의 데이터 이해가 필요해야 함:
     - ID와 이름
     - 타임스탬프와 날짜/시간 (월, 일, 년, 초)
     - 파일 ID, 이름, 확장자, MIME 타입
     - URL, GID 등

9. **질문은 주로 실제 인간 사용 사례를 반영해야 함**
   - LLM의 도움을 받는 인간이 관심을 가질 만한 정보 검색 작업

10. **질문은 수십 번의 도구 호출을 필요로 할 수 있음**
    - 컨텍스트가 제한된 LLM에 도전
    - MCP 서버 도구가 반환하는 정보를 줄이도록 장려

11. **모호한 질문 포함**
    - 모호하거나 어떤 도구를 호출해야 할지 어려운 결정이 필요할 수 있음
    - LLM이 잠재적으로 실수하거나 잘못 해석하도록 유도
    - 모호함에도 불구하고 **단일 검증 가능한 답변이 있어야 함**

### 안정성

12. **질문은 답변이 변하지 않도록 설계해야 함**
    - "현재 상태"에 의존하는 질문 금지
    - 예를 들어 다음을 세지 않기:
      - 게시물에 대한 반응 수
      - 스레드의 답글 수
      - 채널의 멤버 수

13. **MCP 서버가 만드는 질문의 종류를 제한하지 않도록 하기**
    - 도전적이고 복잡한 질문 만들기
    - 일부는 사용 가능한 MCP 서버 도구로 풀기 어려울 수 있음
    - 질문은 특정 출력 형식을 요구할 수 있음 (날짜/시간 vs. 에포크 시간, JSON vs. MARKDOWN)
    - 질문은 완료하는 데 수십 번의 도구 호출을 필요로 할 수 있음

## 답변 가이드라인

### 검증

1. **답변은 직접 문자열 비교로 검증 가능해야 함**
   - 답변이 여러 형식으로 작성될 수 있다면 질문에서 출력 형식을 명확히 지정
   - 예: "YYYY/MM/DD 형식으로 답하세요", "True 또는 False로 답하세요", "A, B, C, D 중 하나로 답하세요"
   - 답변은 단일 검증 가능한 값이어야 함:
     - 사용자 ID, 사용자 이름, 표시 이름, 이름, 성
     - 채널 ID, 채널 이름
     - 메시지 ID, 문자열
     - URL, 제목
     - 숫자 수량
     - 타임스탬프, 날짜/시간
     - 불리언 (참/거짓 질문)
     - 이메일 주소, 전화번호
     - 파일 ID, 파일 이름, 파일 확장자
     - 객관식 답변

### 가독성

2. **답변은 일반적으로 사람이 읽을 수 있는 형식을 선호해야 함**
   - 예: 이름, 날짜/시간, 파일 이름, 메시지 문자열, URL, 예/아니오, 참/거짓, a/b/c/d
   - 불투명한 ID보다 선호 (ID도 허용)
   - 답변의 대다수는 사람이 읽을 수 있어야 함

### 안정성

3. **답변은 안정적/고정적이어야 함**
   - 이전 콘텐츠 (예: 종료된 대화, 출시된 프로젝트, 답변된 질문) 조회
   - 항상 같은 답변을 반환할 "닫힌" 개념 기반으로 질문 생성
   - 비정적 답변을 차단하기 위해 고정된 시간 창을 고려하도록 질문 작성
   - 변경될 가능성이 낮은 컨텍스트 활용
   - 예: 논문 이름을 찾는다면, 나중에 출판된 논문과 혼동되지 않을 만큼 충분히 구체적으로

4. **답변은 명확하고 모호하지 않아야 함**
   - 단일하고 명확한 답변이 있도록 질문 설계
   - MCP 서버 도구를 사용하여 도출 가능한 답변

### 다양성

5. **답변은 다양해야 함**
   - 다양한 형태와 형식의 단일 검증 가능한 값
   - 사용자 개념: 사용자 ID, 사용자 이름, 표시 이름, 이름, 성, 이메일 주소, 전화번호
   - 채널 개념: 채널 ID, 채널 이름, 채널 주제
   - 메시지 개념: 메시지 ID, 메시지 문자열, 타임스탬프, 월, 일, 년

6. **답변은 복잡한 구조가 아니어야 함**
   - 값 목록 안 됨
   - 복잡한 객체 안 됨
   - ID나 문자열 목록 안 됨
   - 자연어 텍스트 안 됨
   - 단, 직접 문자열 비교로 간단히 검증할 수 있고 현실적으로 재현 가능한 경우는 예외

## 평가 프로세스

### 1단계: 문서 검사

대상 API의 문서를 읽어 이해하기:
- 사용 가능한 엔드포인트와 기능
- 모호함이 있으면 웹에서 추가 정보 가져오기
- 이 단계를 가능한 많이 병렬화
- 각 서브에이전트는 파일 시스템이나 웹의 문서만 검사해야 함

### 2단계: 도구 검사

MCP 서버에서 사용 가능한 도구 목록 확인:
- MCP 서버를 직접 검사
- 입출력 스키마, 독스트링, 설명 이해
- 이 단계에서 도구를 직접 호출하지 않기

### 3단계: 이해 심화

충분한 이해를 얻을 때까지 1단계와 2단계 반복:
- 여러 번 반복
- 만들고 싶은 작업 유형 생각
- 이해 정제
- **어떤 단계에서도 MCP 서버 구현 코드 자체를 읽지 않기**
- 직관과 이해를 사용하여 합리적이고 현실적이지만 매우 도전적인 작업 생성

### 4단계: 읽기 전용 콘텐츠 검사

API와 도구를 이해한 후 MCP 서버 도구 사용:
- **읽기 전용이고 비파괴적인 작업만 사용**하여 콘텐츠 검사
- 목표: 현실적인 질문 생성을 위한 특정 콘텐츠 (사용자, 채널, 메시지, 프로젝트, 작업 등) 파악
- 상태를 수정하는 도구는 절대 호출하지 않기
- MCP 서버 구현 코드 자체는 읽지 않기
- 독립적인 탐색을 수행하는 개별 서브에이전트로 이 단계를 병렬화
- **주의**: 일부 도구는 많은 데이터를 반환하여 컨텍스트 부족을 유발할 수 있음
- 탐색을 위해 점진적이고 작고 목표가 명확한 도구 호출 하기
- 모든 도구 호출 요청에서 `limit` 매개변수를 사용하여 결과 제한 (<10)
- 페이지네이션 사용

### 5단계: 작업 생성

콘텐츠 검사 후 10개의 사람이 읽을 수 있는 질문 생성:
- LLM이 MCP 서버를 사용하여 답할 수 있는 질문
- 위의 모든 질문 및 답변 가이드라인 준수

## 출력 형식

각 QA 쌍은 질문과 답변으로 구성됩니다. 출력은 다음 구조의 XML 파일이어야 합니다:

```xml
<evaluation>
   <qa_pair>
      <question>2024년 2분기에 생성된 프로젝트 중 완료된 작업이 가장 많은 것을 찾으세요. 프로젝트 이름은?</question>
      <answer>Website Redesign</answer>
   </qa_pair>
   <qa_pair>
      <question>"bug"로 태그된 이슈 중 2024년 3월에 종료된 것을 검색하세요. 어떤 사용자가 가장 많은 이슈를 종료했나요? 사용자 이름을 제공하세요.</question>
      <answer>sarah_dev</answer>
   </qa_pair>
   <qa_pair>
      <question>/api 디렉토리의 파일을 수정하고 2024년 1월 1일부터 31일 사이에 병합된 풀 리퀘스트를 찾으세요. 이 PR에 참여한 서로 다른 기여자는 몇 명인가요?</question>
      <answer>7</answer>
   </qa_pair>
   <qa_pair>
      <question>2023년 이전에 생성된 저장소 중 별이 가장 많은 것을 찾으세요. 저장소 이름은?</question>
      <answer>data-pipeline</answer>
   </qa_pair>
</evaluation>
```

## 평가 예제

### 좋은 질문 예시

**예시 1: 심층 탐색이 필요한 다중 홉 질문 (GitHub MCP)**
```xml
<qa_pair>
   <question>2023년 3분기에 아카이브된 저장소 중 이전에 조직에서 가장 많이 포크된
   프로젝트를 찾으세요. 해당 저장소에서 사용된 주요 프로그래밍 언어는?</question>
   <answer>Python</answer>
</qa_pair>
```

이 질문이 좋은 이유:
- 아카이브된 저장소를 찾기 위한 여러 검색 필요
- 아카이브 전 가장 많은 포크를 가진 저장소를 식별해야 함
- 언어를 위한 저장소 세부 정보 확인 필요
- 답변이 단순하고 검증 가능한 값
- 변하지 않는 역사적 (닫힌) 데이터 기반

**예시 2: 키워드 매칭 없이 컨텍스트 이해 필요 (프로젝트 관리 MCP)**
```xml
<qa_pair>
   <question>2023년 말에 완료된 고객 온보딩 개선에 초점을 맞춘 이니셔티브를
   찾으세요. 프로젝트 리더가 완료 후 회고 문서를 작성했습니다. 당시
   리더의 직책은?</question>
   <answer>Product Manager</answer>
</qa_pair>
```

이 질문이 좋은 이유:
- 특정 프로젝트 이름을 사용하지 않음
- 특정 시간대의 완료된 프로젝트를 찾아야 함
- 프로젝트 리더와 그 역할을 식별해야 함
- 회고 문서의 컨텍스트 이해 필요
- 답변이 사람이 읽을 수 있고 안정적
- 완료된 작업 기반 (변하지 않음)

**예시 3: 여러 단계가 필요한 복잡한 집계 (이슈 트래커 MCP)**
```xml
<qa_pair>
   <question>2024년 1월에 보고되어 중요 우선순위로 표시된 모든 버그 중,
   할당된 버그를 48시간 이내에 가장 높은 비율로 해결한 담당자는?
   사용자 이름을 제공하세요.</question>
   <answer>alex_eng</answer>
</qa_pair>
```

이 질문이 좋은 이유:
- 날짜, 우선순위, 상태로 버그를 필터링해야 함
- 담당자별로 그룹화하고 해결률 계산해야 함
- 48시간 창을 결정하기 위해 타임스탬프 이해 필요
- 페이지네이션 테스트 (처리할 버그가 많을 수 있음)
- 답변이 단일 사용자 이름
- 특정 기간의 역사적 데이터 기반

**예시 4: 여러 데이터 유형에 걸친 종합 필요 (CRM MCP)**
```xml
<qa_pair>
   <question>2023년 4분기에 Starter에서 Enterprise 요금제로 업그레이드한 계정 중
   연간 계약 가치가 가장 높은 계정을 찾으세요. 이 계정이 속한 산업은?</question>
   <answer>Healthcare</answer>
</qa_pair>
```

### 나쁜 질문 예시

**예시 1: 시간에 따라 답변이 변함**
```xml
<qa_pair>
   <question>현재 엔지니어링 팀에 할당된 미해결 이슈는 몇 개인가요?</question>
   <answer>47</answer>
</qa_pair>
```

나쁜 이유:
- 이슈가 생성, 종료, 재할당됨에 따라 답변이 변함
- 안정적/고정적 데이터 기반이 아님
- 동적인 "현재 상태"에 의존

**예시 2: 키워드 검색으로 너무 쉬움**
```xml
<qa_pair>
   <question>"Add authentication feature" 제목의 풀 리퀘스트를 찾아 누가 만들었는지 알려주세요.</question>
   <answer>developer123</answer>
</qa_pair>
```

나쁜 이유:
- 정확한 제목을 직접 키워드 검색으로 풀 수 있음
- 심층 탐색이나 이해가 필요 없음
- 종합 또는 분석 불필요

**예시 3: 모호한 답변 형식**
```xml
<qa_pair>
   <question>Python이 주 언어인 모든 저장소를 나열하세요.</question>
   <answer>repo1, repo2, repo3, data-pipeline, ml-tools</answer>
</qa_pair>
```

나쁜 이유:
- 답변이 어떤 순서로도 반환될 수 있는 목록
- 직접 문자열 비교로 검증하기 어려움
- LLM이 다르게 포맷할 수 있음 (JSON 배열, 쉼표로 구분, 줄바꿈으로 구분)
- 특정 집계 (수) 또는 최상급 (가장 많은 별)을 묻는 것이 더 좋음

## 검증 프로세스

평가 생성 후:

1. **XML 파일 확인** - 스키마 이해
2. **각 작업 지침 로드** - MCP 서버와 도구를 사용하여 작업을 직접 풀어 올바른 답변 확인 (병렬로)
3. **WRITE 또는 파괴적 작업이 필요한 작업 표시**
4. **모든 올바른 답변 축적** - 문서의 잘못된 답변 교체
5. **WRITE 또는 파괴적 작업이 필요한 `<qa_pair>` 제거**

컨텍스트 부족을 피하기 위해 작업 해결을 병렬화한 다음, 모든 답변을 축적하고 파일 끝에 변경사항 적용하기.

## 좋은 평가 만들기 팁

1. **미리 열심히 생각하고 계획하기** - 작업 생성 전에
2. **기회가 있을 때 병렬화하기** - 속도 향상 및 컨텍스트 관리
3. **현실적인 사용 사례에 집중하기** - 인간이 실제로 달성하고 싶어하는 것
4. **도전적인 질문 만들기** - MCP 서버 기능의 한계를 테스트
5. **안정성 보장하기** - 역사적 데이터와 닫힌 개념 사용
6. **답변 검증하기** - MCP 서버 도구를 사용하여 직접 질문 풀기
7. **반복 및 개선하기** - 프로세스 중 배운 것을 기반으로

# 평가 실행하기

평가 파일을 생성한 후, 제공된 평가 하네스를 사용하여 MCP 서버를 테스트할 수 있습니다.

## 설정

1. **의존성 설치**

   ```bash
   pip install -r scripts/requirements.txt
   ```

   또는 수동 설치:
   ```bash
   pip install anthropic mcp
   ```

2. **API 키 설정**

   ```bash
   export ANTHROPIC_API_KEY=your_api_key_here
   ```

## 평가 파일 형식

평가 파일은 `<qa_pair>` 요소를 사용한 XML 형식입니다:

```xml
<evaluation>
   <qa_pair>
      <question>2024년 2분기에 생성된 프로젝트 중 완료된 작업이 가장 많은 것을 찾으세요. 프로젝트 이름은?</question>
      <answer>Website Redesign</answer>
   </qa_pair>
   <qa_pair>
      <question>"bug"로 태그된 이슈 중 2024년 3월에 종료된 것을 검색하세요. 어떤 사용자가 가장 많은 이슈를 종료했나요?</question>
      <answer>sarah_dev</answer>
   </qa_pair>
</evaluation>
```

## 평가 실행

평가 스크립트(`scripts/evaluation.py`)는 세 가지 전송 방식을 지원합니다:

**중요:**
- **stdio 전송**: 평가 스크립트가 자동으로 MCP 서버 프로세스를 시작하고 관리합니다. 서버를 수동으로 실행하지 마세요.
- **sse/http 전송**: 평가를 실행하기 전에 MCP 서버를 별도로 시작해야 합니다. 스크립트는 지정된 URL에서 이미 실행 중인 서버에 연결합니다.

### 1. 로컬 STDIO 서버

로컬에서 실행되는 MCP 서버 (스크립트가 서버를 자동으로 시작):

```bash
python scripts/evaluation.py \
  -t stdio \
  -c python \
  -a my_mcp_server.py \
  evaluation.xml
```

환경 변수 포함:
```bash
python scripts/evaluation.py \
  -t stdio \
  -c python \
  -a my_mcp_server.py \
  -e API_KEY=abc123 \
  -e DEBUG=true \
  evaluation.xml
```

### 2. SSE (Server-Sent Events)

SSE 기반 MCP 서버 (먼저 서버를 시작해야 함):

```bash
python scripts/evaluation.py \
  -t sse \
  -u https://example.com/mcp \
  -H "Authorization: Bearer token123" \
  -H "X-Custom-Header: value" \
  evaluation.xml
```

### 3. HTTP (Streamable HTTP)

HTTP 기반 MCP 서버 (먼저 서버를 시작해야 함):

```bash
python scripts/evaluation.py \
  -t http \
  -u https://example.com/mcp \
  -H "Authorization: Bearer token123" \
  evaluation.xml
```

## 명령줄 옵션

```
사용법: evaluation.py [-h] [-t {stdio,sse,http}] [-m MODEL] [-c COMMAND]
                     [-a ARGS [ARGS ...]] [-e ENV [ENV ...]] [-u URL]
                     [-H HEADERS [HEADERS ...]] [-o OUTPUT]
                     eval_file

위치 인수:
  eval_file             평가 XML 파일 경로

선택적 인수:
  -h, --help            도움말 메시지 표시
  -t, --transport       전송 방식: stdio, sse, 또는 http (기본: stdio)
  -m, --model           사용할 Claude 모델 (기본: claude-3-7-sonnet-20250219)
  -o, --output          보고서 출력 파일 (기본: stdout에 출력)

stdio 옵션:
  -c, --command         MCP 서버 실행 명령 (예: python, node)
  -a, --args            명령에 대한 인수 (예: server.py)
  -e, --env             KEY=VALUE 형식의 환경 변수

sse/http 옵션:
  -u, --url             MCP 서버 URL
  -H, --header          'Key: Value' 형식의 HTTP 헤더
```

## 출력

평가 스크립트는 다음을 포함한 자세한 보고서를 생성합니다:

- **요약 통계**:
  - 정확도 (정답/전체)
  - 평균 작업 시간
  - 작업당 평균 도구 호출 수
  - 총 도구 호출 수

- **작업별 결과**:
  - 프롬프트와 기대 응답
  - 에이전트의 실제 응답
  - 답변 정확 여부 (✅/❌)
  - 시간과 도구 호출 세부 정보
  - 에이전트의 접근 방식 요약
  - 도구에 대한 에이전트의 피드백

## 문제 해결

### 연결 오류

연결 오류가 발생하면:
- **STDIO**: 명령과 인수가 올바른지 확인
- **SSE/HTTP**: URL이 접근 가능하고 헤더가 올바른지 확인
- 필요한 API 키가 환경 변수나 헤더에 설정되어 있는지 확인

### 낮은 정확도

많은 평가가 실패하면:
- 각 작업에 대한 에이전트의 피드백 검토
- 도구 설명이 명확하고 포괄적인지 확인
- 입력 매개변수가 잘 문서화되어 있는지 확인
- 도구가 너무 많거나 너무 적은 데이터를 반환하는지 고려
- 오류 메시지가 실행 가능한지 확인

### 타임아웃 문제

작업이 타임아웃되면:
- 더 강력한 모델 사용 (예: `claude-3-7-sonnet-20250219`)
- 도구가 너무 많은 데이터를 반환하는지 확인
- 페이지네이션이 올바르게 작동하는지 확인
- 복잡한 질문 단순화 고려
