# MCP 서버 모범 사례

## 빠른 참조

### 서버 이름 지정
- **Python**: `{서비스}_mcp` (예: `slack_mcp`)
- **Node/TypeScript**: `{서비스}-mcp-server` (예: `slack-mcp-server`)

### 도구 이름 지정
- snake_case + 서비스 접두사 사용
- 형식: `{서비스}_{동작}_{대상}`
- 예: `slack_send_message`, `github_create_issue`

### 응답 형식
- JSON과 Markdown 형식 모두 지원
- JSON: 프로그래밍 처리용
- Markdown: 사람이 읽기 위한 형식

### 페이지네이션
- 항상 `limit` 매개변수 존중
- `has_more`, `next_offset`, `total_count` 반환
- 기본값 20-50개

### 전송 방식
- **Streamable HTTP**: 원격 서버, 다중 클라이언트 시나리오
- **stdio**: 로컬 통합, 명령줄 도구
- SSE는 사용하지 말 것 (Streamable HTTP로 대체됨)

## 서버 이름 규칙

다음 표준화된 이름 패턴을 따르세요:

**Python**: `{서비스}_mcp` 형식 사용 (소문자 + 밑줄)
- 예: `slack_mcp`, `github_mcp`, `jira_mcp`

**Node/TypeScript**: `{서비스}-mcp-server` 형식 사용 (소문자 + 하이픈)
- 예: `slack-mcp-server`, `github-mcp-server`, `jira-mcp-server`

이름은 일반적이고, 통합되는 서비스를 설명적으로 나타내며, 작업 설명에서 쉽게 추론 가능해야 하며, 버전 번호나 날짜를 포함하지 않아야 합니다.

## 도구 이름 지정 및 설계

### 도구 이름 규칙

1. **snake_case 사용**: `search_users`, `create_project`, `get_channel_info`
2. **서비스 접두사 포함**: MCP 서버가 다른 MCP 서버와 함께 사용될 수 있으므로
   - `send_message` 대신 `slack_send_message` 사용
   - `create_issue` 대신 `github_create_issue` 사용
3. **동작 중심**: 동사로 시작 (get, list, search, create 등)
4. **구체적으로**: 다른 서버와 충돌할 수 있는 일반적인 이름 피하기

### 도구 설계

- 도구 설명은 기능을 좁고 명확하게 기술해야 함
- 설명은 실제 기능과 정확히 일치해야 함
- 도구 어노테이션 제공 (readOnlyHint, destructiveHint, idempotentHint, openWorldHint)
- 도구 작업은 집중적이고 원자적으로 유지

## 응답 형식

데이터를 반환하는 모든 도구는 여러 형식을 지원해야 합니다:

### JSON 형식 (`response_format="json"`)
- 기계가 읽을 수 있는 구조화된 데이터
- 사용 가능한 모든 필드와 메타데이터 포함
- 일관된 필드 이름과 타입
- 프로그래밍 처리에 사용

### Markdown 형식 (`response_format="markdown"`, 보통 기본값)
- 사람이 읽을 수 있는 포맷된 텍스트
- 헤더, 목록, 서식을 사용하여 명확하게
- 타임스탬프를 사람이 읽을 수 있는 형식으로 변환
- 표시 이름과 함께 ID를 괄호 안에 표시
- 상세한 메타데이터 생략

## 페이지네이션

리소스를 나열하는 도구의 경우:

- **항상 `limit` 매개변수를 존중**
- **페이지네이션 구현**: `offset` 또는 커서 기반 페이지네이션 사용
- **페이지네이션 메타데이터 반환**: `has_more`, `next_offset`/`next_cursor`, `total_count` 포함
- **모든 결과를 메모리에 로드하지 않기**: 대규모 데이터셋에 특히 중요
- **합리적인 기본 제한**: 20-50개가 일반적

페이지네이션 응답 예시:
```json
{
  "total": 150,
  "count": 20,
  "offset": 0,
  "items": [...],
  "has_more": true,
  "next_offset": 20
}
```

## 전송 옵션

### Streamable HTTP

**적합한 경우**: 원격 서버, 웹 서비스, 다중 클라이언트 시나리오

**특성**:
- HTTP를 통한 양방향 통신
- 여러 동시 클라이언트 지원
- 웹 서비스로 배포 가능
- 서버에서 클라이언트로의 알림 가능

**사용 시기**:
- 여러 클라이언트에 동시 서비스 제공
- 클라우드 서비스로 배포
- 웹 애플리케이션과 통합

### stdio

**적합한 경우**: 로컬 통합, 명령줄 도구

**특성**:
- 표준 입출력 스트림 통신
- 간단한 설정, 네트워크 설정 불필요
- 클라이언트의 서브프로세스로 실행

**사용 시기**:
- 로컬 개발 환경을 위한 도구 구축
- 데스크톱 애플리케이션과 통합
- 단일 사용자, 단일 세션 시나리오

**참고**: stdio 서버는 stdout에 로그를 출력하면 안 됩니다 (로깅에는 stderr 사용)

### 전송 방식 선택 가이드

| 기준 | stdio | Streamable HTTP |
|------|-------|-----------------|
| **배포** | 로컬 | 원격 |
| **클라이언트** | 단일 | 다중 |
| **복잡도** | 낮음 | 중간 |
| **실시간** | 아니오 | 예 |

## 보안 모범 사례

### 인증 및 권한 부여

**OAuth 2.1**:
- 인증된 기관의 인증서를 사용하는 안전한 OAuth 2.1 사용
- 요청을 처리하기 전에 액세스 토큰 검증
- 서버에 특별히 의도된 토큰만 수락

**API 키**:
- API 키는 환경 변수에 저장, 코드에 절대 포함하지 않기
- 서버 시작 시 키 검증
- 인증 실패 시 명확한 오류 메시지 제공

### 입력 검증

- 디렉토리 탐색을 방지하기 위한 파일 경로 검증
- URL 및 외부 식별자 검증
- 매개변수 크기 및 범위 확인
- 시스템 호출에서 명령 주입 방지
- 모든 입력에 스키마 검증 (Pydantic/Zod) 사용

### 오류 처리

- 내부 오류를 클라이언트에 노출하지 않기
- 보안 관련 오류는 서버 측에서 로깅
- 도움이 되지만 내부 정보를 드러내지 않는 오류 메시지
- 오류 발생 후 리소스 정리

### DNS 리바인딩 보호

로컬에서 실행되는 Streamable HTTP 서버의 경우:
- DNS 리바인딩 보호 활성화
- 모든 수신 연결에서 `Origin` 헤더 검증
- `0.0.0.0` 대신 `127.0.0.1`에 바인딩

## 도구 어노테이션

클라이언트가 도구 동작을 이해할 수 있도록 어노테이션을 제공하세요:

| 어노테이션 | 타입 | 기본값 | 설명 |
|-----------|------|--------|------|
| `readOnlyHint` | boolean | false | 도구가 환경을 수정하지 않음 |
| `destructiveHint` | boolean | true | 도구가 파괴적 업데이트를 수행할 수 있음 |
| `idempotentHint` | boolean | false | 같은 인수로 반복 호출해도 추가 효과 없음 |
| `openWorldHint` | boolean | true | 도구가 외부 엔티티와 상호작용 |

**중요**: 어노테이션은 힌트이며, 보안 보장이 아닙니다. 클라이언트는 어노테이션만을 기반으로 보안에 중요한 결정을 내려서는 안 됩니다.

## 오류 처리

- 표준 JSON-RPC 오류 코드 사용
- 도구 오류는 결과 객체 내에서 보고 (프로토콜 수준 오류가 아닌)
- 유용하고 구체적인 오류 메시지 + 다음 단계 제안
- 내부 구현 세부사항 노출하지 않기
- 오류 발생 시 리소스 적절히 정리

오류 처리 예시:
```typescript
try {
  const result = performOperation();
  return { content: [{ type: "text", text: result }] };
} catch (error) {
  return {
    isError: true,
    content: [{
      type: "text",
      text: `오류: ${error.message}. filter='active_only'를 사용하여 결과를 줄여보세요.`
    }]
  };
}
```

## 테스트 요구사항

포괄적인 테스트 항목:

- **기능 테스트**: 유효/유효하지 않은 입력으로 올바른 실행 확인
- **통합 테스트**: 외부 시스템과의 상호작용 테스트
- **보안 테스트**: 인증, 입력 검증, 속도 제한 검증
- **성능 테스트**: 부하, 타임아웃 상태에서의 동작 확인
- **오류 처리**: 적절한 오류 보고 및 정리 보장
